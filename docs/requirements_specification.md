# Stream Player Web 要件定義書（仕様書）

**文書バージョン**: 2.0
**作成日**: 2026-03-04
**最終更新日**: 2026-03-05
**対象バージョン**: Web版初期リリース（React + FastAPI + Docker 構成）

---

## 1. アプリケーション概要・目的

### 1.1 概要

Stream Player Web は、YouTube等のストリーミングコンテンツを読み込み、任意の時間位置にマーカーを配置し、マーカー間のセグメントを定義してループ再生を行う **Webアプリケーション** である。さらにテンポ（再生速度）とピッチ（音程）の独立制御が可能で、音楽練習・語学学習・動画分析等の用途に適する。

**アーキテクチャ**: フロントエンド（React + TypeScript）とバックエンドAPI（Python FastAPI）をDockerコンテナで構成し、ブラウザ上で動作する。

### 1.2 目的

- ストリーミング動画の特定区間を繰り返し再生し、効率的な学習・練習を支援する
- テンポ変更時にピッチを維持し、ピッチ変更時にテンポを維持する独立制御を提供する
- マーカー・セグメント・エフェクト設定をURL単位でブラウザのlocalStorageに自動保存し、再訪問時に復元する

### 1.3 想定ユーザー

- 楽器練習者（特定フレーズの低速ループ練習）
- 語学学習者（リスニング区間の反復再生）
- 動画分析者（特定シーンの繰り返し確認）

## 2. 機能要件

### 2.1 ストリーミング再生機能

| ID | 機能 | 詳細 |
|----|------|------|
| F-PLAY-001 | URL読み込み | YouTube等のストリーミングURLを入力し、動画を読み込む。フロントエンドで video ID を抽出できない場合は、バックエンドの `/api/resolve` エンドポイント経由で yt-dlp によりメタデータ（タイトル・長さ）を取得する |
| F-PLAY-002 | 動画再生 | YouTube IFrame Player API による動画・音声の再生。React コンポーネント内に IFrame を埋め込み表示する |
| F-PLAY-003 | 再生/一時停止 | トグル操作で再生と一時停止を切り替える |
| F-PLAY-004 | 停止 | 再生を完全に停止する |
| F-PLAY-005 | シーク（絶対位置） | タイムライン上のクリックまたは数字キー(0-9)で動画の任意の位置にジャンプする。数字キーは動画長の0%〜90%に対応する |
| F-PLAY-006 | シーク（相対位置） | 現在位置から前後に移動する。±1秒、±5秒、±10秒の3段階を提供する |
| F-PLAY-007 | フレームステップ | 一時停止中に微小単位（約0.04秒）で前進・後退する（※ YouTube IFrame API の制約により厳密なフレーム単位ではなく近似） |
| F-PLAY-008 | 音量調整 | 音量を0〜100の範囲で±5単位で調整する |

### 2.2 URL履歴機能

| ID | 機能 | 詳細 |
|----|------|------|
| F-HIST-001 | 履歴保存 | 読み込み成功したURLとタイトルを localStorage の `url_history` キーに保存する |
| F-HIST-002 | 履歴表示 | ドロップダウンメニューで過去のURLを一覧表示する。タイトルがある場合はタイトルを表示し、60文字を超える場合は末尾を省略する |
| F-HIST-003 | 履歴選択 | ドロップダウンから選択すると、URL入力欄にURLが自動入力される |
| F-HIST-004 | 履歴上限 | 最大50件を保持し、最新のものが先頭に表示される。同一URLが重複する場合は古い方を削除する |

| F-MKR-001 | マーカー追加 | 現在の再生位置にマーカーを追加する。ラベルは自動でA, B, C, ..., Z, AA, AB, ...と連番付与される |
| F-MKR-002 | マーカー削除 | 指定したマーカーを削除する。該当マーカーを参照するセグメントも連動して削除される |
| F-MKR-003 | マーカー全削除 | 全マーカーを一括削除する。ラベルカウンターもリセットされる |
| F-MKR-004 | マーカー位置変更 | タイムライン上でマーカーをドラッグして位置を変更する。ドラッグ中は `ew-resize` カーソルを表示する |
| F-MKR-005 | マーカーメモ | 各マーカーにテキストメモを付与できる。`blur` イベントまたは Enter キーで保存される |
| F-MKR-006 | ラベルスワップ | 2つのマーカーのラベル（表示名）を入れ替える。内部IDは変更されない |
| F-MKR-007 | マーカーへジャンプ | マーカーパネルのシークボタンでそのマーカー位置に再生位置を移動する |

| F-MKR-009 | タイムライン表示 | マーカーはタイムラインコンポーネント上に色付き三角形（SVG / Canvas）とラベルテキストで表示される |

#### マーカーデータ構造（TypeScript 型定義）

```typescript
interface Marker {
  id: string;        // UUID (12文字hex)。不変の内部識別子
  label: string;     // 表示ラベル (A, B, ..., AA, AB, ...)
  position: number;  // 秒単位の時間位置
  color: string;     // hex色コード
  memo: string;      // ユーザーメモ（任意）
}
```

### 2.4 セグメント・シーケンスループ機能

| ID | 機能 | 詳細 |
|----|------|------|
| F-SEQ-001 | セグメント定義 | 開始マーカーと終了マーカーを選択してセグメント（区間）を定義する。同一マーカー同士のセグメントは作成不可 |
| F-SEQ-002 | セグメント名 | 各セグメントに任意の表示名を付与できる。未指定の場合は開始マーカーのメモから自動補完される |
| F-SEQ-003 | バッチ追加 | テキスト入力で複数セグメントを一括追加する |
| F-SEQ-004 | セグメント削除 | 個別のセグメントをリストから削除する |
| F-SEQ-005 | セグメント並び替え | 上下ボタンでセグメントの順序を変更する |
| F-SEQ-006 | ループ開始 | シーケンスの先頭セグメントから順に再生を開始する。一時停止中の場合は自動で再生を開始する |
| F-SEQ-007 | ループ停止 | ループ再生を停止する（通常再生に戻る） |
| F-SEQ-008 | セグメントジャンプ | セグメントリストの行クリックで特定セグメントの開始位置にジャンプし、そこからループを継続する |
| F-SEQ-009 | アクティブセグメント表示 | 現在再生中のセグメントをリスト上でハイライト表示し、タイムライン上で半透明の青色領域として表示する |
| F-SEQ-010 | 境界検出 | `setInterval` による再生位置ポーリング（250ms間隔）で、再生位置がセグメント終了位置の0.15秒前に達した時点で次セグメントへ遷移する |

#### ループモード

| モード | 内部値 | 動作 |
|--------|--------|------|
| Loop Sequence | `loop_sequence` | 全セグメントを順番に繰り返す（末尾→先頭に戻る） |
| Loop Single | `loop_single` | 現在のセグメントのみを無限に繰り返す |
| Play Once | `play_once` | 全セグメントを1回ずつ再生し、最後まで到達したら停止する |

#### バッチ入力フォーマット（F-SEQ-003）

```
入力形式:
  AB,CD,EF          -- カンマ区切り
  AB CD EF           -- スペース区切り
  AB/CD/EF           -- スラッシュ区切り
  AB:intro,CD:verse  -- コロンでセグメント名を指定
  混在も可: AB:intro CD/EF

解析ルール:
  - 大文字小文字を区別しない（自動で大文字化）
  - 1文字ラベル: "AB" → (A, B)
  - 多文字ラベル: "AAAB" → 既知マーカーラベルに一致する分割点を探索
  - 両方のラベルが既存マーカーに存在しない場合はスキップ
```

#### セグメントデータ構造（TypeScript 型定義）

```typescript
interface Segment {
  startMarkerId: string;  // 開始マーカーのID
  endMarkerId: string;    // 終了マーカーのID
  displayName: string;    // 表示名（任意）
}

// セグメントの実際の時間範囲は、2つのマーカー位置のmin/maxで正規化される
// （開始マーカーが終了マーカーより後ろでも正常動作する）
```

### 2.7 オーディオエフェクト機能

| ID | 機能 | 詳細 |
|----|------|------|
| F-EFX-001 | テンポ変更 | 再生速度を0.25x〜2.0xの範囲で変更する。YouTube IFrame API の `setPlaybackRate()` を使用。ピッチは YouTube 側で自動的に維持される |
| F-EFX-002 | テンポプリセット | 0.5x, 0.75x, 1.0x, 1.25x, 1.5x, 2.0x のプリセットボタンを提供する。現在のテンポに一致するボタンがハイライトされる |
| F-EFX-003 | テンポスライダー | 0.25x〜2.0x を `<input type="range">` スライダーで微調整する |
| F-EFX-004 | トランスポーズ | 音程を-12〜+12半音の範囲でシフトする。テンポは維持される |
| F-EFX-005 | トランスポーズスライダー | -12〜+12を24ステップ（1半音刻み）の `<input type="range">` スライダーで調整する |
| F-EFX-006 | トランスポーズ±ボタン | ±1半音単位で音程を変更するボタンを提供する |
| F-EFX-007 | エフェクトリセット | テンポを1.0x、トランスポーズを0半音にリセットする |
| F-EFX-008 | UI自動同期 | キーボードショートカットやスライダー操作のいずれからエフェクトが変更されても、Reactの状態管理により全UIコンポーネントが自動同期する |

#### トランスポーズ技術仕様（Web版）

```
実装: Web Audio API (AudioWorklet)
  - YouTube IFrame から MediaStream を取得し AudioContext に接続
  - PitchShifterNode（AudioWorklet）でリアルタイムピッチシフト
  - 計算式: pitchRatio = 2^(semitones / 12)

※ YouTube IFrame API の CORS 制約により、AudioContext への接続が
  ブロックされる場合がある。その場合はフォールバックとして
  テンポ変更のみを提供し、ピッチ制御は将来対応とする。

0半音の場合: ピッチシフトをバイパス（パススルー）
```

### 2.8 永続化機能

| ID | 機能 | 詳細 |
|----|------|------|
| F-SAVE-001 | 自動保存 | マーカーまたはセグメントが変更されると、2秒のデバウンス後に localStorage へ自動保存される |
| F-SAVE-002 | URL切り替え時保存 | 新しいURLを読み込む際、現在のURLの設定を即座に保存する |
| F-SAVE-003 | ページ離脱時保存 | `beforeunload` イベントで現在の設定を保存する |
| F-SAVE-004 | 設定復元 | URLの読み込み時、localStorage から以前保存した設定（マーカー・指定プリセットのセグメント・ループモード）を復元する |
| F-SAVE-005 | 空データ自動削除 | マーカーとセグメントが両方とも空の場合、そのURLの設定エントリを localStorage から削除する |
| F-SAVE-006 | URL正規化 | 保存キーとしてURLを正規化する。一時的なクエリパラメータ（`t`, `feature`, `si`）を除去し、残りのパラメータをソートする |

#### 永続化ストレージ（localStorage キー）

| キー | 内容 | 形式 |
|------|------|------|
| `stream_player_loop_settings` | URL別のマーカー・プリセット（セグメント・ループモード） | JSON |
| `stream_player_url_history` | URL履歴（URL + タイトル） | JSON |
| `stream_player_playlist` | プレイリスト（URL・タイトル・プリセット名の配列） | JSON |

#### loop_settings データフォーマット（v2.0）

```json
{
  "normalized_url": {
    "markers": [
      {"id": "xxxx", "label": "A", "position": 10.5, "color": "#FF6B6B", "memo": ""}
    ],
    "presets": {
      "default": {
        "segments": [{"startMarkerId": "...", "endMarkerId": "...", "displayName": ""}],
        "loopMode": "loop_sequence"
      }
    }
  }
}
```

- マーカーはプリセット間で共有される（同一URL = 同一マーカーセット）
- セグメント構成とループモードのみがプリセットごとに異なる
- キー名はフロントエンドの TypeScript 型定義に合わせて camelCase

#### playlist データフォーマット

```json
[
  {"url": "https://...", "title": "動画タイトル", "preset": "default"},
  {"url": "https://...", "title": "別の動画", "preset": "イントロ練習"}
]
```

#### ショートカット制御ルール

- `<input>`, `<textarea>`, `<select>` 等のフォーム要素にフォーカスがある場合（`document.activeElement.tagName` で判定）、全てのキーボードショートカットは無効化される
- Escape キーとダブルクリックはフォーカス状態に関係なく動作する
- ブラウザのデフォルト動作は `e.preventDefault()` で抑制する

| 再生遅延 | YouTube IFrame API の読み込みに依存（通常1〜3秒） |
| バックエンド解決 | `/api/resolve` のレスポンスは yt-dlp 解決時間に依存（通常2〜5秒） |
| セグメント遷移 | 境界検出閾値 0.15秒以内で次セグメントにシーク |
| 自動保存 | 変更後2秒のデバウンスで localStorage に書き込み。UIをブロックしない |
| タイムライン更新 | `setInterval` による250ms間隔のポーリングで再生位置を更新 |
| 初回ロード | フロントエンドバンドルサイズは gzip 圧縮後 200KB 以下を目標 |

**設計原則**:
- hooks/ と components/ はReactの単方向データフローで接続
- stores/ はlocalStorage の読み書きを抽象化し、コンポーネントから直接 localStorage を操作しない
- マーカーは不変のUUID(ID)で管理し、ラベルは表示専用

- バックエンドはステートレス。状態はすべてフロントエンド側で管理する

### 4.3 テスト観点メモ（QAエンジニア向け）

以下はテスト計画策定時に特に注意すべき観点:

1. **デバウンス動作**: マーカーの連続追加・削除時に自動保存が正しくデバウンスされるか（2秒間操作がなければ1回だけ localStorage に保存）
2. **URL正規化の一貫性**: 同一動画の異なるURLバリエーション（`&t=10`, `&feature=share`, `&si=xxx`付き等）が同じ保存キーに正規化されるか
3. **セグメント境界の正確性**: 0.15秒閾値での境界検出が、テンポ変更時（再生速度が変わる）にも正しく動作するか
4. **マーカー削除の連鎖**: マーカー削除→参照セグメント削除→React 状態更新→タイムライン再描画の一連の連鎖が正しく完了するか
5. **復元時の保存抑制**: 設定復元中に自動保存がトリガーされないこと（`isRestoring` フラグによるガード）
6. **プリセット切替の保存整合性**: プリセットA→B切替時にAの変更が保存され、Bの設定が正しく復元されること。マーカーは共有のまま維持されること
7. **Play Once自動遷移**: プレイリスト内でPlay Once完了→次動画読み込み→プリセット復元→自動開始の一連のフローがエラーなく完了すること
8. **プレイリスト終端処理**: プレイリスト最後の動画でPlay Once完了時、無限ループせずに停止すること
9. **defaultプリセット削除防止**: UIから「default」プリセットの削除を試みた場合、何も起きないこと
10. **localStorage 容量**: 大量のURL設定が蓄積された場合の localStorage 容量（通常5MB上限）への対応
11. **ブラウザ互換性**: Chrome / Firefox / Edge / Safari で主要機能が動作すること
12. **バックエンド API**: `/api/resolve` が無効URL入力時に適切なHTTPステータス（400）とエラーメッセージを返すこと
13. **Docker 環境**: `docker compose up --build` で frontend / backend が正常起動し、Vite プロキシ経由で API 通信できること
